# Wikix — LLM-Curated Personal Wiki

**Inspired by:** [github.com/akshajsrivastava-exe/wikix](https://github.com/akshajsrivastava-exe/wikix)  
**Source:** GitHub Search / LLM personal wiki (early stage, 0 stars)  
**Interest score:** 3/4 (dev productivity/logging + Claude/LLM tooling + daily utility)

## What It Is

Wikix is an LLM agent that ingests raw sources — links, PDFs, pasted notes, snippets — and curates them into a structured, cross-linked personal wiki: it extracts key points, files each source under the right topic, updates existing pages instead of duplicating, and maintains `[[wikilinks]]` between related notes. The value over a plain note app is the **agent doing the filing and linking** so your knowledge base stays organized without manual gardening.

## Why Build Your Own

The reference is bare. What fills a real gap: a local, markdown-on-disk wiki (Obsidian-compatible) where dropping a URL or PDF triggers Claude to (a) summarize, (b) decide which existing page it belongs to or create a new one, and (c) wire cross-links. Owning it means plain `.md` files you keep forever, no proprietary store, and a taxonomy Claude adapts to *your* existing notes rather than a fixed schema.

## Stack Recommendation

- **Node.js + TypeScript** CLI/daemon — watches an `inbox/` folder
- **Claude API (`claude-sonnet-5`)** — summarize, classify, link
- **Markdown files on disk** — `wiki/*.md` with YAML frontmatter, Obsidian-compatible `[[links]]`
- **`@mozilla/readability` + `jsdom`** — clean article text from URLs
- **`pdf-parse`** — extract PDF text
- **`chokidar`** — watch the inbox for new drops
- No database — the filesystem *is* the store

## MVP Scope

Drop a `.url`, `.pdf`, or `.md` file into `inbox/`. A watcher extracts the text, asks Claude for a summary + the best-fit existing page (from a list of current page titles) or a new page name, appends a dated, sourced entry to that page, and inserts `[[links]]` to related pages Claude names. You edit the wiki in Obsidian; Wikix just keeps it tidy.

## Implementation Phases

### Phase 1 — Source Extraction
**Goal:** Turn a URL/PDF/note into clean plain text.

**Files:**
- `src/extract.ts` — URL / PDF / markdown extractors
- `inbox/` — watched drop folder

**Key steps:**
1. `pnpm add @mozilla/readability jsdom pdf-parse chokidar`
2. `.url` file (single line) → fetch → Readability → main text + title
3. `.pdf` → `pdf-parse` → text
4. `.md`/`.txt` → read as-is
5. Normalize to `{title, text, sourceUrl?, addedAt}`

**Verify:** Drop a blog-post URL → clean article text + title printed, boilerplate stripped.

---

### Phase 2 — Summarize + Classify
**Goal:** Claude decides where each source belongs.

**Files:**
- `src/curate.ts` — Claude prompt for summary + target page
- `src/wiki.ts` — read existing page titles

**Key steps:**
1. Gather current `wiki/*.md` titles as the candidate topic list
2. Prompt: "Summarize this source in 3–5 bullets. Given these existing pages [list], pick the single best page to add it to, or propose a new page title if none fit. Return JSON `{summary, page, isNew, relatedPages[]}`."
3. Validate JSON with Zod; retry once on malformed output
4. Log the decision

**Verify:** Drop an article about Postgres indexing → returns `{page: "PostgreSQL", isNew: false, ...}` when that page exists.

---

### Phase 3 — Write + Cross-Link
**Goal:** Append to the target page and wire `[[links]]`.

**Files:**
- extend `src/wiki.ts` — append entry, create page, insert links

**Key steps:**
1. If `isNew`, create `wiki/<Page>.md` with frontmatter (`created`, `tags`)
2. Append a dated section: `## <date> — <source title>` + summary bullets + `Source: <url>`
3. For each `relatedPage` that exists, ensure a `[[relatedPage]]` link appears in a "Related" section
4. Move the processed file from `inbox/` to `inbox/done/`

**Verify:** After two related articles, both pages exist and reference each other via `[[links]]`; open the vault in Obsidian and see the graph connect them.

---

### Phase 4 — Watch Daemon + Dedup
**Goal:** Continuous, idempotent ingestion.

**Files:**
- `src/watch.ts` — chokidar loop
- `src/dedup.ts` — content hash check

**Key steps:**
1. Watch `inbox/`; process each new file, then move to `done/`
2. Hash source text; skip if the same source was already ingested (avoid dupes)
3. Serialize processing (queue) so two drops don't edit the same page concurrently
4. Write failures to `inbox/failed/` with an error note

**Verify:** Drop the same URL twice → second is skipped as a duplicate; concurrent drops don't corrupt a page.

---

### Phase 5 — Ask-Your-Wiki Query
**Goal:** Question the knowledge base in natural language.

**Files:**
- `src/ask.ts` — retrieve relevant pages + answer

**Key steps:**
1. `wikix ask "what do I know about vector databases?"`
2. Keyword/title match to shortlist pages (no embeddings needed at MVP scale); load their text
3. Claude answers citing which pages/sources it used
4. Optionally expose the same as an MCP tool so Claude Code can consult the wiki mid-task

**Verify:** After ingesting a few sources, `wikix ask` returns an answer that cites the correct pages.

---

## Estimated Effort

**1 Claude session.** Phases 1–3 (extract → curate → write) are the core ~2.5 hours and already give you a self-organizing wiki. Watch daemon + query add ~1.5 hours. No DB and markdown-on-disk keep it small.

## Potential Blockers

- **Claude picks the wrong page / spawns near-duplicates** ("PostgreSQL" vs "Postgres"): pass the exact existing titles and instruct exact-match reuse; periodically ask Claude to propose page merges rather than letting the taxonomy sprawl.
- **Concurrent writes to one page**: two inbox drops targeting the same page can clobber each other. Serialize processing through a single queue (Phase 4) — don't parallelize file edits.
- **Paywalled/JS-heavy URLs**: Readability fails on many modern pages. Detect empty extractions and route to `failed/` with a note instead of filing garbage.
- **Cost on large PDFs**: a 200-page PDF is a lot of tokens. Chunk + cap input, or summarize section-by-section, and warn when a source exceeds a size threshold.
- **Idempotency**: without the dedup hash, re-scanning the inbox re-appends entries. Build dedup in Phase 4 before running it as a daemon on real data.
