# Kojo — Mobile Cron for Your Coding Agents

**Inspired by:** [github.com/loppo-llc/kojo](https://github.com/loppo-llc/kojo)  
**Source:** GitHub Search / Claude agent CLI (v0.110.0, MIT, 4 stars)  
**Interest score:** 4/4 (Claude/LLM tooling + agent UI + dev productivity + mobile-first)

## What It Is

Kojo runs a small server on your dev machine that exposes your terminal AI coding CLIs (Claude Code, Codex) to a mobile web UI. Beyond a phone terminal, it adds **persistent agents** — named sessions with memory that can be **scheduled** to run autonomously and message each other. The reference build is heavy (Go PTY, Tailscale P2P, encrypted SQLite, multi-device clustering); the interesting, unique part is "kick off a coding-agent run from your phone, and let it run on a cron even when you're away."

## Why Build Your Own

The reference bundles clustering, an encrypted credentials vault, and Tailscale — most of which you don't need for a single-user setup. The MVP you actually want is smaller: a phone-friendly terminal to one Claude Code session + a scheduler that runs a saved prompt against your repo on a cron and pushes you the result. Owning it means no Tailscale dependency (plain WebSocket over an existing tunnel) and prompts/schedules you control.

## Stack Recommendation

- **Node.js + `node-pty`** — spawn and stream a Claude Code process (simpler than Go's ConPTY handling)
- **Fastify + `ws`** — WebSocket bridge between PTY and browser
- **xterm.js + Tailwind** — mobile-first terminal UI, PWA-installable
- **`node-cron`** — scheduled agent runs
- **SQLite (`better-sqlite3`)** — sessions, saved prompts, run history
- **Web Push (VAPID)** — notify phone when a scheduled run finishes
- Reach the server via an existing tunnel (Cloudflare Tunnel / Tailscale) — don't rebuild networking

## MVP Scope

Open a PWA on your phone → see a live terminal attached to a Claude Code session on your laptop. Save a prompt ("run the test suite and fix failures") as a scheduled job. At the scheduled time the server runs it headless (`claude -p`), captures output, and Web-Push-notifies you with a summary you can open on the phone.

## Implementation Phases

### Phase 1 — PTY ↔ WebSocket Bridge
**Goal:** Stream a shell/Claude Code session to a browser terminal.

**Files:**
- `server/index.ts` — Fastify + WS server
- `server/pty.ts` — `node-pty` spawn + I/O piping
- `web/terminal.html` — xterm.js client

**Key steps:**
1. `pnpm add fastify ws node-pty better-sqlite3`
2. On WS connect, `pty.spawn('claude', [], {cols, rows})`; pipe pty→ws and ws→pty
3. xterm.js `term.onData` → send to ws; ws message → `term.write`
4. Handle resize messages (cols/rows) to keep formatting correct on mobile
5. Serve the page as a PWA (manifest + service worker) so it installs to the home screen

**Verify:** On your phone (same tunnel), type in the terminal and see Claude Code respond live.

---

### Phase 2 — Session Persistence
**Goal:** Sessions survive disconnects/crashes so a dropped mobile connection doesn't kill the agent.

**Files:**
- `server/sessions.ts` — session registry keyed by id
- `db/schema.sql` — `sessions`, `runs` tables

**Key steps:**
1. Keep PTY processes alive server-side in a `Map<id, pty>`; WS clients attach/detach
2. Buffer recent output (ring buffer, last ~2000 lines) and replay on reattach
3. Persist session metadata to SQLite; list active sessions in the UI
4. Add a "new session" / "kill session" control

**Verify:** Start a long Claude task, close the phone browser, reopen — output is intact and streaming continues.

---

### Phase 3 — Scheduled Agent Runs
**Goal:** Run a saved prompt against a repo on a cron, headless.

**Files:**
- `server/scheduler.ts` — `node-cron` jobs
- `web/jobs.html` — create/list scheduled jobs

**Key steps:**
1. Job = `{name, cwd, prompt, cron}` stored in SQLite
2. On fire: `execFile('claude', ['-p', prompt], {cwd})`, capture stdout/exit code into `runs`
3. Cap runtime (e.g. 10 min) and output size; mark failures
4. UI to add a job (prompt textarea + cron picker) and view last run per job

**Verify:** Schedule "summarize today's git diff" for 1 minute from now → a `runs` row appears with the summary.

---

### Phase 4 — Push Notifications
**Goal:** Get pinged on your phone when a scheduled run finishes.

**Files:**
- `server/push.ts` — Web Push (VAPID) send
- `web/sw.js` — service worker `push` handler

**Key steps:**
1. Generate VAPID keys once; subscribe the PWA and store the subscription
2. On run completion, push `{title, body: first 120 chars of output, jobId}`
3. Tapping the notification opens the run detail view
4. Add a per-job toggle for notify-on-success vs notify-on-failure-only

**Verify:** Lock your phone, wait for a scheduled run → notification arrives; tapping it opens the output.

---

### Phase 5 — Auth + Saved Prompt Library
**Goal:** Lock down access and make re-running common tasks one tap.

**Files:**
- `server/auth.ts` — single shared token (bearer) check on WS + HTTP
- `web/library.html` — saved prompt snippets

**Key steps:**
1. Require a bearer token (from env) on every connection; reject otherwise
2. Saved prompts library ("fix lint", "write missing tests", "update changelog") — tap to run now or schedule
3. Rate-limit run creation to avoid runaway loops
4. Optional: per-repo default cwd picker

**Verify:** Without the token, connection is refused. With it, tap a saved prompt → runs against the selected repo.

---

## Estimated Effort

**1.5 Claude sessions.** Phases 1–3 (terminal bridge + persistence + scheduler) are the core ~3 hours. Phases 4–5 add ~1.5 hours. Skipping Tailscale/clustering/encryption from the reference is what keeps this a weekend build.

## Potential Blockers

- **Exposing a shell to the network is dangerous**: require a bearer token from the first byte, bind to localhost, and reach it only through an authenticated tunnel — never open the port publicly.
- **`node-pty` native build**: needs build tools; pin Node version and document `npm rebuild` for the deploy box.
- **Headless `claude -p` auth**: the scheduled path needs a valid Claude Code session/API key in the server environment; verify non-interactive auth works before building the scheduler UI.
- **Runaway scheduled loops**: a job that edits+commits could loop. Hard-cap runtime and output, and default schedules to notify-only (no auto-push) until you trust a job.
- **Mobile keyboard vs terminal**: on-screen keyboards lack Ctrl/Esc/arrows. Add an accessory key bar (Esc, Tab, Ctrl, ↑↓) above the terminal — do this in Phase 1 or the UI is unusable.
