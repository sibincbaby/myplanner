## Overview

mcpsnoop is a single binary that intercepts the stdio channel between an MCP client (Claude Code, Cursor, Codex, any MCP consumer) and an MCP server. It forwards every byte unchanged while printing each JSON-RPC frame ŌĆö requests, responses, notifications, and errors ŌĆö to a live terminal UI or log file.

**Why it matters:** MCP's stdio transport is a black box. The Inspector connects separately so it never sees real client messages. Server stderr is sparse and timing-imprecise. mcpsnoop is the only tool that observes the actual negotiation and tool-call sequence your agent makes under production conditions.

**Key capabilities from the upstream repo:**
- Transparent passthrough: client and server both think they're talking directly to each other.
- Live terminal output: colour-coded JSON-RPC frames with timestamps and latency.
- Slow/hung call detection: configurable timeout marks stalled calls visually.
- Capture-and-replay: recorded sessions can be replayed against the server for regression testing.
- Query filtering: filter displayed frames by method name or token substring.
- Zero config: install one binary, prepend it to the command string in your MCP config.

## Stack Recommendation

For a personal build inspired by mcpsnoop:

- **Language:** Go (same as upstream) ŌĆö `os.Stdin`/`os.Stdout` piping is clean, no runtime to install, single-binary output.

## MVP Scope

**In scope:**
- Spawn the real MCP server as a subprocess from `argv[1:]`.
- Pipe stdin ŌåÆ child-stdin, child-stdout ŌåÆ stdout (the transparent passthrough).
- Tee both streams to stderr (or a `--log` file) with timestamps and direction arrows (`ŌåÆ` / `ŌåÉ`).
- Pretty-print each complete JSON-RPC frame (newline-delimited JSON detection).
- Mark frames that exceed a configurable `--slow` threshold (default 2s) with a warning.

**Out of scope for MVP:** TUI, capture-replay, query filtering, capability diff view, multi-server support.

## Implementation Phases

### Phase 1: Transparent stdio proxy
**Goal:** `mcpsnoop npx -y @modelcontextprotocol/server-filesystem /tmp` behaves identically to running the server directly, but logs every frame to stderr.

- `cmd/mcpsnoop/main.go`: parse `os.Args[1:]` as the server command; `exec.Command(args[0], args[1:]...)` with `cmd.Stdin = os.Stdin`, `cmd.Stdout = os.Stdout`, `cmd.Stderr = os.Stderr`.
- Replace `cmd.Stdin`/`cmd.Stdout` with pipe pairs; launch two goroutines: one copies stdinŌåÆchild-stdin while writing each chunk to a channel, one copies child-stdoutŌåÆstdout while writing each chunk to a channel.
- A third goroutine reads from both channels and writes direction-tagged bytes to stderr.
- `cmd.Wait()` at the end; propagate exit code.

**Verify:** replace the filesystem MCP server entry in `claude_desktop_config.json` with `["mcpsnoop", "npx", "-y", "@modelcontextprotocol/server-filesystem", "/tmp"]`; open Claude Desktop; confirm tools work and stderr shows raw frames.

### Phase 2: JSON-RPC frame detection + pretty-printing
**Goal:** Each complete JSON-RPC object is pretty-printed with a timestamp instead of showing raw byte chunks.

- Buffer incoming bytes per direction; scan for complete newline-delimited JSON objects.
- `json.Unmarshal` each candidate; on success, `json.MarshalIndent` and print with direction arrow, timestamp, and method/id fields highlighted.
- Partial/binary data (non-JSON) falls through as a hex dump.

**Verify:** stderr shows well-formatted frames like:
```
ŌåÆ 12:34:56.789 [request #1] tools/call
   {"jsonrpc":"2.0","id":1,"method":"tools/call","params":{...}}
ŌåÉ 12:34:57.012 [response #1] +223ms
   {"jsonrpc":"2.0","id":1,"result":{...}}
```

### Phase 3: Slow-call detection + `--log` flag
**Goal:** Frames that take longer than `--slow N` ms are marked; all output can be redirected to a file.

- Track pending request IDs with a `sync.Map[id]time.Time`; on response, compute latency and print a `ŌÜĀ SLOW` marker if over threshold.
- Add `--log <path>` flag: open the file and `io.MultiWriter(os.Stderr, logFile)` for all output.
- Add `--slow <ms>` flag (default 2000).

### Phase 4 (optional): Capture + replay
**Goal:** Record a session to a JSONL file; replay it against the server to regression-test changes.

- `--capture <file>`: write each frame as a JSONL line with direction, timestamp, and raw JSON.
- `mcpsnoop replay <capture-file> <server-command>`: read requests from the file, send them to a fresh server subprocess, compare responses against recorded ones.

## Potential Blockers

- **Buffering breaks the MCP handshake:** if the proxy buffers too aggressively, the client and server may stall waiting for bytes. Ensure the stdinŌåÆchild-stdin goroutine copies in real-time (no intermediate buffer larger than the read chunk; avoid `bufio.Writer` without explicit `Flush`).
- **Binary / non-JSON-RPC servers:** some MCP servers may use binary framing or emit non-JSON stderr; the frame parser must not crash on partial or malformed data. Wrap `json.Unmarshal` in a `recover()` and fall back to hex dump.
- **Exit code propagation:** if the MCP server exits with a non-zero code, the proxy must propagate it. `cmd.Wait()` returns `*exec.ExitError`; extract `ExitCode()` and `os.Exit()` with it.
- **Claude Desktop config syntax:** the config value must be an array of strings (not a shell string), so `["mcpsnoop", "npx", "-y", "...", "/tmp"]` is correct; a single string with spaces will break. Document this clearly.
