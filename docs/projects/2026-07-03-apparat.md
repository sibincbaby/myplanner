# Apparat — Voice-Driven Agentic Git

**Inspired by:** [github.com/cjtrowbridge/apparat](https://github.com/cjtrowbridge/apparat)  
**Source:** GitHub Search / voice LLM agent (personal-area-network agentic git manager, early stage)  
**Interest score:** 3/4 (dev productivity + voice tools + Claude/LLM tooling)

## What It Is

Apparat is a personal-area-network agent that manages git repos with **local speech-to-text and text-to-speech plus an LLM** — you talk to your repositories and it acts. The reference frames it as a broad PAN "operator". The sharp, buildable core: a voice loop where you say "commit these changes with a message about the auth fix", "what changed since yesterday", or "create a branch for the invoice bug", and a Claude-backed agent runs the git and talks the result back. Hands-free dev-log and git driving.

## Why Build Your Own

The reference is ambitious and vague (whole personal-area network). What fills a real gap for you is narrow: a local voice → intent → git/dev-command loop that speaks results back, running fully on your machine. Owning it lets you wire it to *your* repos and your safe-command allowlist, and pick local Whisper (privacy) over a cloud STT. It doubles as a spoken dev diary — every voice session logged to a daily markdown file.

## Stack Recommendation

- **Python + `faster-whisper`** — local STT (no audio leaves the machine)
- **`piper` (or macOS `say`)** — local TTS for spoken responses
- **Claude API (`claude-sonnet-5`)** — turn a transcript into a structured tool call
- **`sounddevice`** — mic capture with push-to-talk / VAD
- **`GitPython` + `subprocess`** — run git and whitelisted shell commands
- **Markdown log** — append each turn to `journal/YYYY-MM-DD.md`

## MVP Scope

Push-to-talk on the terminal → local Whisper transcribes → Claude maps the transcript to one of a small allowlisted tool set (`git_status`, `git_diff`, `git_commit`, `git_branch`, `git_log`, `run_tests`) with args → execute → summarize the result → speak it back via TTS, and append the exchange to today's journal file. Destructive actions require a spoken "yes" confirmation.

## Implementation Phases

### Phase 1 — Local Voice Loop
**Goal:** Speak → text → speak, fully local, no LLM yet.

**Files:**
- `apparat/audio.py` — mic capture (push-to-talk) + `faster-whisper` transcribe
- `apparat/tts.py` — `piper`/`say` wrapper
- `apparat/main.py` — loop: record → transcribe → echo via TTS

**Key steps:**
1. `pip install faster-whisper sounddevice`
2. Push-to-talk: hold a key to record, release to transcribe (`small.en` model)
3. Speak the transcript back to confirm STT quality
4. Measure round-trip latency; pick model size for <2s on your hardware

**Verify:** Say "hello world, this is a test" → it prints and speaks the same text within ~2s.

---

### Phase 2 — Transcript → Structured Tool Call
**Goal:** Claude converts free speech into one validated command.

**Files:**
- `apparat/agent.py` — Claude call with tool schema
- `apparat/tools.py` — tool definitions (name + JSON args)

**Key steps:**
1. Define tools: `git_status`, `git_diff`, `git_commit{message}`, `git_branch{name}`, `git_log{n}`, `run_tests`
2. Send transcript + tool schema to Claude with `tool_choice: auto`; parse the returned tool + args
3. If Claude returns no tool (ambiguous), speak a clarifying question instead of guessing
4. Log `{transcript, tool, args}` for debugging

**Verify:** "commit everything with a message about fixing the login bug" → parses to `git_commit{message:"Fix login bug"}`.

---

### Phase 3 — Safe Execution + Spoken Results
**Goal:** Run the command against the real repo and speak a summary.

**Files:**
- `apparat/exec.py` — GitPython/subprocess runners, per-tool
- extend `main.py` — execute → summarize → TTS

**Key steps:**
1. Allowlist only the defined tools; reject anything else (never `eval` a raw shell string)
2. Run in the configured repo `cwd`; capture stdout/stderr
3. Send the raw output back to Claude for a one-sentence spoken summary
4. Speak it; on error, speak the error briefly

**Verify:** "what's changed" → runs `git status`, speaks "You have 3 modified files and 1 untracked file."

---

### Phase 4 — Confirmation Gate for Destructive Actions
**Goal:** No commit/branch/push without an explicit spoken yes.

**Files:**
- `apparat/confirm.py` — confirmation state machine

**Key steps:**
1. Classify tools as read (auto-run) vs write (`git_commit`, `git_branch`, `run_tests` if it mutates)
2. For write actions, speak "About to commit with message X — confirm?" and wait for a yes/no utterance
3. Timeout → cancel; anything but a clear "yes" → cancel
4. Log the confirmed action

**Verify:** "commit this" → it asks to confirm; say "no" → nothing happens; say "yes" → it commits.

---

### Phase 5 — Spoken Dev Journal
**Goal:** Every session becomes a searchable daily log.

**Files:**
- `apparat/journal.py` — append exchanges to `journal/YYYY-MM-DD.md`

**Key steps:**
1. Append `- HH:MM — "<transcript>" → <tool> → <result summary>` per turn
2. A `summarize today` voice command asks Claude to read the day's file and speak a recap
3. Optional: end-of-day auto-summary written to the top of the file
4. Keep journals in a git repo of their own for history

**Verify:** After a few voice commands, `journal/2026-07-03.md` contains timestamped entries; "summarize today" speaks an accurate recap.

---

## Estimated Effort

**1.5 Claude sessions.** Phase 1 (local voice) is the fiddliest — ~1.5 hours to get latency and push-to-talk right. Phases 2–4 (agent + safe exec + confirmation) are ~2 hours. The journal is ~30 min.

## Potential Blockers

- **Never execute arbitrary shell from a transcript**: only dispatch to the fixed allowlisted tool functions with validated args. A voice agent that can run any command is a foot-gun aimed at your repo.
- **STT mishears destructive intent**: "commit" vs "revert" vs "reset" sound distinct but errors happen — the Phase 4 spoken confirmation is non-negotiable for any write, and echo the parsed action before executing.
- **Whisper latency on CPU**: `small.en` may be slow without a GPU. Benchmark early; drop to `base.en` or `tiny.en` if the loop feels laggy, and use VAD to avoid transcribing silence.
- **Mic/audio device setup on Linux**: `sounddevice`/PortAudio can be finicky. Verify capture works standalone before wiring the pipeline.
- **Ambient noise false triggers**: prefer explicit push-to-talk over always-listening VAD for the MVP — a hot mic that acts on overheard speech is worse than a key press.
