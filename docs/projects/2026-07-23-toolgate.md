# ToolGate

> BM25-based context engineering layer that pre-selects the right tools before each Claude Code turn — cuts token overhead and fixes tool-overload accuracy loss, no vector DB required

**Inspired by:** [ratel-ai/ratel](https://github.com/ratel-ai/ratel) (306 ★)  
**Date discovered:** 2026-07-23

---

## What gap it fills

Claude Code's system prompt gets stuffed with every configured MCP tool schema on every turn. At 20+ tools the total schema payload can burn 8–15k tokens before your prompt even starts, and accuracy drops because the model gets confused choosing between dozens of irrelevant options. ToolGate sits in front of tool dispatch: it reads the user's message, runs a lightweight BM25 search over the tool schemas, and injects only the top-K relevant tools into the context. No embeddings, no inference call on the retrieval path — deterministic and instant.

## Stack recommendation

| Layer | Choice | Why |
|-------|--------|-----|
| Core search | BM25 (wink-bm25-text-search or rank-bm25 in Python) | Zero infrastructure, in-process, no vector DB |
| Wrapper | TypeScript MCP server | Directly hooks into Claude Code's MCP pipeline |
| Tool catalog store | JSON file on disk | Simple, git-trackable, no DB |
| Config | `.toolgate.json` per project | Top-K, min-score threshold, always-include list |
| Optional | Cohere re-ranker API | Single API call for semantic re-ranking when BM25 isn't enough |

## MVP scope (1 Claude session)

One MCP server that proxies another MCP server's tool catalog through BM25 filtering:

1. **`toolgate-server.ts`**: MCP server that, on `tools/list`, fetches the downstream catalog from a target MCP server and caches it; on each `tools/call`, re-ranks the catalog against the current user query (passed as metadata) and returns only top-K tool schemas
2. **`.toolgate.json`**: `{ "topK": 8, "minScore": 0.1, "alwaysInclude": ["bash", "read"] }`
3. **CLI**: `npx toolgate init` that scaffolds the config and patches `~/.claude/settings.json`

Out of scope for MVP: semantic re-ranker, multi-server chaining, per-turn usage telemetry.

## Phases

### Phase 1 — BM25 index build (1 h)
- On startup, fetch `tools/list` from each configured downstream server
- Concatenate tool name + description + parameter names into a text document per tool
- Build in-memory BM25 index with wink-bm25-text-search
- Cache index; rebuild on schema change (hash comparison)

### Phase 2 — Query-time filtering (1 h)
- Intercept MCP `tools/list` calls
- Extract last user message from MCP request context or env var `TOOLGATE_QUERY`
- Run BM25 query → top-K results → return only those tool schemas
- Fall through: if query is empty or BM25 returns 0 hits, return full catalog

### Phase 3 — Always-include + min-score (0.5 h)
- Read `.toolgate.json`
- Merge `alwaysInclude` list into every filtered response regardless of score
- Drop tools below `minScore` even if top-K would include them

### Phase 4 — Claude Code integration (0.5 h)
- `npx toolgate init`: writes MCP server entry into `~/.claude/settings.json`
- `npx toolgate status`: prints catalog size, index size, last query and top-K chosen

### Phase 5 — Token savings logging (0.5 h)
- Count schema bytes before and after filter
- Log `[toolgate] 23 → 7 tools, saved ~4,200 tokens` to stderr on each turn
- Optional: append JSONL telemetry to `~/.toolgate/log.jsonl`

## Effort estimate

~2.5 hours for Phases 1–3 (MVP) · 1 Claude session  
~4 hours complete with CLI and telemetry

## Blockers / risks

- **Query access**: MCP protocol doesn't always surface the current user message at `tools/list` time. Mitigation: read `TOOLGATE_QUERY` env var that a Claude Code pre-tool hook sets; or proxy the entire `messages/create` call to intercept the latest human message.
- **Schema drift**: Downstream MCP servers update their tool catalogs; stale BM25 index gives wrong results. Mitigation: hash the raw tool list on each `tools/list` request and rebuild if changed — BM25 index build is sub-millisecond for <200 tools.
- **Always-include list maintenance**: Users forget to update it. Mitigation: auto-detect core tools (bash, read, write) by checking schema names against a known-safe list.
