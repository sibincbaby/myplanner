## Overview

frugon answers a question every heavy agent user eventually asks: *how much of my LLM bill was waste?* It sits in front of your API calls as a proxy, logs them verbatim, then re-prices the captured workload against alternative models — showing the delta if you'd routed cheaper.

**Two components:**
- **`frugon capture`** — a localhost proxy that forwards requests to the real provider unchanged and appends `{request, response, model, tokens, timestamp}` to a JSONL log.
- **`frugon analyze <log>`** — reads the log, tokenizes each prompt locally, prices it against a set of candidate models, and reports total spend + projected savings per routing strategy.

Upstream reports 30–50% savings on mixed workloads, up to 85% on repetitive tasks — plausible because a lot of agent traffic is cheap, formulaic calls sent to an expensive model by default.

## Stack Recommendation

For a personal build:

- **Language:** Python (matches upstream; `pipx install` / `uvx` distribution).
- **Proxy:** stdlib `http.server` or a tiny `aiohttp` pass-through — log then forward, never transform.
- **Tokenizer:** provider-appropriate local tokenizer (e.g. `tiktoken` for OpenAI-shaped, a HF tokenizer for others) — approximate is fine for cost estimates.
- **Pricing:** a hand-maintained `pricing.toml` (per-model input/output $/Mtok); refresh occasionally — this is the one thing that drifts.
- **Optional `[measure]` extra:** replay a sample of calls against the cheaper model with your own key to score quality, so savings aren't recommended blindly.

## MVP Scope

**In scope:**
- `frugon capture --port 8788 --upstream https://api.anthropic.com` — pass-through proxy writing JSONL.
- `frugon analyze logs.jsonl` — total spend, per-model breakdown, and "if routed to <cheaper>" savings projection.
- `pricing.toml` covering the models you actually use.
- A one-line summary table: `Model | Calls | Spend | Cheaper option | Est. saving`.

**Out of scope for MVP:** automatic live routing (this only *recommends*), a web dashboard, streaming-response reassembly beyond token counting, non-OpenAI-shaped API decoding.

## Implementation Phases

### Phase 1: Capture proxy
**Goal:** point a client at `localhost:8788`, traffic flows normally, JSONL grows.

- `capture.py`: async pass-through; on each call, record model, prompt, completion, token counts (from the response usage field when present), timestamp; forward untouched.
- Config: upstream base URL + port via flags/env.

**Verify:** run one real agent session through it; confirm responses are byte-identical and the JSONL has one line per call.

### Phase 2: Local re-pricing analyzer
**Goal:** `frugon analyze` prints spend + savings.

- `pricing.toml`: input/output $/Mtok for each candidate model.
- `analyze.py`: for each logged call, compute actual cost; compute counterfactual cost under each cheaper candidate; aggregate; render the summary table.
- Local tokenizer fallback when the response omits usage counts.

### Phase 3: Quality sampling (optional `[measure]`)
**Goal:** don't recommend a downgrade that would've produced worse output.

- Sample N calls, replay against the recommended cheaper model with the user's key, and flag ones where output diverges materially (length/embedding-distance heuristic).
- Report savings *net* of calls that failed the quality bar.

## Estimated Effort

**1–2 Claude Code sessions.** Phase 1 (proxy) + Phase 2 (analyzer) is a complete, useful tool in one session. Phase 3 is a second session and only worth it if you'll act on the recommendations.

## Potential Blockers

- **Pricing drift:** model prices change; a stale `pricing.toml` silently skews every number. Stamp the file with a "prices as of" date and surface it in the report.
- **Streaming responses:** SSE bodies need reassembly to count output tokens; for MVP, prefer the provider's `usage` field and treat streaming calls as best-effort.
- **Tokenizer mismatch:** a local tokenizer won't exactly match the provider's billed count — fine for relative comparison, but label estimates as approximate.
- **Proxy as a bottleneck / secret handling:** the proxy sees full prompts and API keys. Keep logs `chmod 600`, never log the `Authorization` header, and make the log path explicit so it's not committed by accident.
