## Overview

peek-cli closes the visual feedback loop for coding agents. Today, when Claude Code edits a component, it's blind to the rendered result — you screenshot and paste. peek-cli makes "look at the page" a tool the agent calls itself.

**Architecture (three tiny pieces):**
- **Chrome extension** — on request, captures the visible area of the active tab and pushes the PNG over a local WebSocket. Deliberately capture-only: it cannot click, type, or navigate, so there's no automation attack surface.
- **CLI/WS server** — `peeked` runs a localhost WebSocket broker between the extension and the agent skill.
- **Claude Code skill** (`view-browser-tab`) — the agent invokes it, receives a screenshot, and reasons over it.

## Stack Recommendation

For a personal build:

- **Extension:** plain Manifest V3, `chrome.tabs.captureVisibleTab()` — no framework needed.
- **Bridge:** Node + `ws` (single file). Or Bun for a smaller footprint.
- **Agent integration:** a Claude Code skill (`.claude/skills/view-browser-tab/`) that shells out to a small client which asks the broker for the latest frame and writes it to a temp PNG the agent reads.
- **Transport:** localhost-only WS on a fixed port; the extension and skill both connect to it.

## MVP Scope

**In scope:**
- Chrome extension: a single "capture active tab" message handler over WS.
- Local WS broker: relay a capture request → return base64 PNG.
- Claude Code skill: `view-browser-tab` — request frame, save to temp file, return path for the agent to read.
- Handle the "no extension connected" case with a clear error the agent can relay.

**Out of scope for MVP:** full-page (scrolling) capture, element-scoped capture, multi-tab selection, Firefox port, any interaction beyond screenshots.

## Implementation Phases

### Phase 1: WS broker + extension capture
**Goal:** clicking the extension icon prints a PNG to disk via the broker.

- `broker.js`: `ws` server on `localhost:8787`; two message types — `capture:request`, `capture:frame`.
- `extension/`: MV3 manifest with `activeTab` + `<all_urls>` capture permission; background service worker connects to the broker and answers `capture:request` with `captureVisibleTab()`.

**Verify:** trigger a capture, confirm a valid PNG lands in `/tmp`.

### Phase 2: Claude Code skill
**Goal:** `Show me the current tab` inside Claude Code returns a screenshot the agent describes.

- `.claude/skills/view-browser-tab/SKILL.md` + a small client that sends `capture:request`, waits for `capture:frame`, writes the PNG, and returns the path.
- Skill description tuned so the agent reaches for it after UI edits.

### Phase 3: Robustness + ergonomics
**Goal:** it fails gracefully and installs in one line.

- Clear errors: no extension connected, tab not capturable (chrome:// pages), timeout.
- `npm i -g` packaging + a `/plugin marketplace add` path for the skill.
- Optional: crop to the viewport DPR so Retina captures aren't 2× oversized (token cost).

## Estimated Effort

**1 Claude Code session.** Phases 1–2 are the whole product; Phase 3 is polish. The security model (screenshot-only) keeps scope naturally small.

## Potential Blockers

- **Chrome capture permissions:** `captureVisibleTab()` needs `activeTab` or host permission and only works on the focused tab — document that chrome:// and PDF viewer tabs can't be captured.
- **Image token cost:** full-resolution Retina PNGs are large; downscale or JPEG-encode before handing to the agent to keep context cheap. Leave a quality knob.
- **WS port collisions:** fixed port may clash; fall back through a small port range and write the chosen port to a dotfile both sides read.
- **Skill discovery:** the agent only uses it if the skill description clearly signals "use after frontend changes" — iterate on that wording.
