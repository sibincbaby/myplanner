## Overview

`agents-ui` is a locally-run Nuxt 4 dashboard that provides a visual GUI over the `~/.claude` directory, so Claude Code config artifacts — agents, commands, skills, and plugins — can be browsed, created, edited, and organized without hand-editing markdown and JSON. The user's real `~/.claude` tree confirms the shape: `agents/*.md` and `commands/*.md` are single markdown files with YAML frontmatter (e.g. `name`, `description`, `mcpServers`), `skills/<name>/SKILL.md` are folder-based skills with frontmatter, and `plugins/` holds JSON config (`config.json`, `installed_plugins.json`) plus `enabledPlugins` toggles in `settings.json`. The app runs as `localhost` only (it reads and writes the user's home directory), using Nuxt's Nitro server routes for all filesystem access and a Vue frontend for editing. This is a self-hosted personal tool, not a deployed multi-user app, so there is no auth, database, or external API — just typed CRUD over a known directory layout.

## Stack Recommendation

- **Nuxt 4** (`^4`) — core stack, SSR disabled where useful but Nitro server routes are the point. Scaffold with `npm create nuxt@latest agents-ui`.
- **Nitro server routes** (`server/api/**`) — all filesystem reads/writes; never touch `fs` from the client.
- **gray-matter** (`^4`) — parse/stringify YAML frontmatter + markdown body. Battle-tested, round-trips cleanly.
- **Nuxt UI** (`@nuxt/ui`, based on Tailwind + Reka UI) — tables, forms, modals, toasts out of the box; fastest path to a clean dashboard. Alternative: plain Tailwind if you want zero opinion.
- **CodeMirror 6** (`codemirror`, `@codemirror/lang-markdown`, `@codemirror/lang-yaml`) — markdown/frontmatter editor pane with syntax highlighting. Lighter than Monaco for this.
- **zod** (`^3`) — validate frontmatter shape and API request bodies server-side.
- **VueUse** (`@vueuse/nuxt`) — debounced autosave, `useFetch` helpers.
- **Config**: `nuxt.config.ts` with `ssr: true` fine (local), and a single `CLAUDE_DIR` env var (default `os.homedir() + '/.claude'`) so tests can point at a fixture dir.

Node note: **requires Node 20+** — the machine currently has v12.22.9, which will not run Nuxt 4 (see Blockers).

## MVP Scope

In scope:
- Left-nav dashboard with four sections: **Agents**, **Commands**, **Skills**, **Plugins**.
- Agents & Commands: list, view, create, edit (frontmatter fields + markdown body via CodeMirror), delete. Frontmatter shown as a structured form (name, description, and free-form extra keys) plus raw body editor.
- Skills: list folder-based skills, read/edit their `SKILL.md` (frontmatter + body), create a new skill folder scaffold, delete a skill folder.
- Plugins: read `installed_plugins.json`, `config.json`, and `enabledPlugins` from `settings.json`; render as a toggle list; enable/disable writes back to `settings.json`.
- Safety: automatic timestamped backup of any file before overwrite (there is already a `~/.claude/backups/` dir to mirror), and a confirm dialog on delete.

Out of scope (MVP): running/testing agents, marketplace install, git integration, multi-directory (project-level `.claude`) support, diff view. These are Phase-5+ stretch.

## Implementation Phases

### Phase 1: Scaffold + read-only filesystem API
**Goal:** The app boots at `localhost:3000` and lists real agents, commands, and skills read live from `~/.claude`.
**Files to create/modify:**
- `nuxt.config.ts` — enable `@nuxt/ui`, `@vueuse/nuxt`; define `runtimeConfig.claudeDir` defaulting to `~/.claude`.
- `server/utils/claude-dir.ts` — resolves base dir from `runtimeConfig`/env, exposes `paths.agents`, `paths.commands`, `paths.skills`, `paths.plugins`, `paths.settings`; guards against path traversal (reject names containing `/` or `..`).
- `server/utils/md-file.ts` — `readMdFile(path)` and helpers using `gray-matter` to return `{ frontmatter, body, raw }`.
- `server/api/agents/index.get.ts` — list `agents/*.md`, return `[{ name, description, path }]`.
- `server/api/commands/index.get.ts` — list `commands/*.md`.
- `server/api/skills/index.get.ts` — list `skills/*/SKILL.md`, return `[{ name, description, dir }]`.
- `app/pages/index.vue` + `app/layouts/default.vue` — left nav + section stubs.
- `app/pages/agents/index.vue`, `app/pages/commands/index.vue`, `app/pages/skills/index.vue` — list tables via `useFetch`.
**Key steps:**
1. `npm create nuxt@latest agents-ui`, then `npm i gray-matter zod && npx nuxi module add ui @vueuse/nuxt`.
2. Implement `claude-dir.ts` reading `useRuntimeConfig().claudeDir`; add `.env` support with `CLAUDE_DIR`.
3. Implement `md-file.ts` using `matter(raw)`; tolerate files with no frontmatter (commands like `git.md` have partial/nonstandard frontmatter — never throw, fall back to `{ frontmatter: {}, body: raw }`).
4. Build the three `index.get.ts` list endpoints with `readdir` + frontmatter parse; sort by name.
5. Build `layouts/default.vue` with a `UNavigationMenu` sidebar and pages that render a `UTable` of results.
**Verify:** `npm run dev`, open `localhost:3000/agents` and see `personal-pa` and `prd-analyzer`; `/commands` shows 20 rows; `/skills` shows the 19 skill folders. `curl localhost:3000/api/agents` returns JSON.

### Phase 2: View + edit agents and commands (write path with backups)
**Goal:** Any agent or command can be opened, edited (frontmatter form + body), and saved back to disk with an automatic backup.
**Files to create/modify:**
- `server/utils/md-file.ts` — add `writeMdFile(path, { frontmatter, body })` that `matter.stringify(body, frontmatter)` and `backupFile(path)` copying to `~/.claude/backups/<name>.<timestamp>.bak` before overwrite.
- `server/utils/validate.ts` — zod schemas for agent/command frontmatter (`name` required string, `description` optional string, passthrough for extra keys like `mcpServers`, `Purpose`).
- `server/api/agents/[name].get.ts` / `[name].put.ts` — read one / write one.
- `server/api/commands/[name].get.ts` / `[name].put.ts`.
- `app/components/FrontmatterForm.vue` — renders known fields as inputs + an "advanced" raw-YAML textarea for extra keys.
- `app/components/MarkdownEditor.vue` — CodeMirror 6 wrapper (client-only, `<ClientOnly>`).
- `app/pages/agents/[name].vue`, `app/pages/commands/[name].vue` — edit views with debounced/explicit Save.
**Key steps:**
1. `npm i codemirror @codemirror/lang-markdown @codemirror/lang-yaml`.
2. Implement `backupFile` + `writeMdFile`; ensure `matter.stringify` preserves body verbatim (round-trip test a real file first to confirm no reformatting surprises).
3. Add `[name].get/put` endpoints; validate body with zod, return 400 on invalid frontmatter.
4. Build `MarkdownEditor.vue` inside `<ClientOnly>` to avoid SSR CodeMirror errors; two-way bind via `v-model`.
5. Wire edit page: load via `useFetch`, edit, Save button `POST/PUT`, toast on success, and reload list.
**Verify:** Edit `personal-pa` description in the UI, Save, then `git diff` (or `cat`) the file to confirm the change and that a `.bak` appeared in `~/.claude/backups/`. Re-open the page and confirm it shows the new value.

### Phase 3: Create + delete for agents, commands, and skills
**Goal:** New agents/commands/skills can be scaffolded from the UI and unwanted ones deleted safely.
**Files to create/modify:**
- `server/api/agents/index.post.ts`, `server/api/agents/[name].delete.ts` (and command equivalents) — create with a starter frontmatter+body template; delete after backup.
- `server/api/skills/[name].get.ts` / `[name].put.ts` — read/write `skills/<name>/SKILL.md`.
- `server/api/skills/index.post.ts` — create `skills/<name>/SKILL.md` folder scaffold with `name`/`description` frontmatter.
- `server/api/skills/[name].delete.ts` — remove skill folder (backup the folder as a tar or copy first).
- `app/pages/skills/[name].vue` — skill edit view (reuse `FrontmatterForm` + `MarkdownEditor`).
- `app/components/NewItemModal.vue` — shared "create" dialog (name + description), validates unique name.
**Key steps:**
1. Add create endpoints; reject if target already exists (409) and re-run the traversal guard on the user-supplied name (slugify to `[a-z0-9-]`).
2. Add delete endpoints; always `backupFile`/copy the target before `rm`, then a confirm modal in the UI (`useConfirm` / `UModal`).
3. Build skill read/write reusing `md-file.ts` pointed at `skills/<name>/SKILL.md`.
4. Wire `NewItemModal` into each list page's "New" button; navigate to the edit page on create.
**Verify:** Create a test agent `zzz-scratch` from the UI, confirm the file exists with correct frontmatter, edit and save it, then delete it and confirm the file is gone and a backup exists. Repeat for a skill folder.

### Phase 4: Plugins view + enable/disable toggles
**Goal:** All installed plugins are listed with an on/off toggle that reads and writes `enabledPlugins` in `settings.json`.
**Files to create/modify:**
- `server/utils/json-file.ts` — safe read/write of JSON with backup + pretty-print (2-space) to preserve `settings.json` formatting.
- `server/api/plugins/index.get.ts` — merge `plugins/installed_plugins.json` (installed set + scopes) with `enabledPlugins` from `settings.json`; return `[{ id, marketplace, scope, enabled }]`.
- `server/api/plugins/[id]/toggle.post.ts` — flip `enabledPlugins[id]` in `settings.json`, backup first.
- `app/pages/plugins/index.vue` — table of plugins with `USwitch` per row.
**Key steps:**
1. Parse `installed_plugins.json` (v2 format: keyed by `plugin@marketplace` with an array of scope entries) into a flat list.
2. Read `enabledPlugins` from `settings.json` (may be absent — default to treating unlisted as enabled/disabled per Claude Code's actual semantics; confirm by inspecting a project that uses `ccplug`).
3. Implement `toggle.post.ts`: read settings JSON, set/unset the key, backup, write with `JSON.stringify(obj, null, 2)`.
4. Render toggles; optimistic UI update with rollback on API error.
**Verify:** Toggle a plugin off in the UI, `grep enabledPlugins ~/.claude/settings.json` to confirm the key changed, confirm a settings backup was written, toggle it back on.

### Phase 5: Polish, guardrails, and docs
**Goal:** The dashboard is safe to run daily — traversal-proof, backup-audited, with search and a README.
**Files to create/modify:**
- `server/middleware/local-only.ts` — reject requests whose host is not `localhost`/`127.0.0.1` (defense against DNS-rebind since this exposes the home dir).
- `app/composables/useSearch.ts` + nav search box — client-side filter across all sections.
- `app/components/BackupsPanel.vue` + `server/api/backups/index.get.ts` — list recent backups so edits are recoverable.
- `README.md` — run instructions, `CLAUDE_DIR` override, safety notes.
- `tests/` — Vitest tests for `md-file.ts` round-trip and API endpoints against a fixture `.claude` dir.
**Key steps:**
1. Add the local-only middleware and verify traversal guards reject `../` in every dynamic route.
2. Add global search filtering the lists.
3. Add a backups list view so the user can see/restore prior versions.
4. Write Vitest tests pointing `CLAUDE_DIR` at `tests/fixtures/.claude`; cover create/edit/delete/toggle happy paths + a traversal-rejection case.
5. Write the README.
**Verify:** `npm run test` passes; `curl 'localhost:3000/api/agents/..%2f..%2fsettings'` is rejected; searching "personal" filters the agent list; README steps reproduce a clean run.

## Potential Blockers

- **Node version (hard blocker, must fix first):** the machine has **Node v12.22.9**; Nuxt 4 requires Node **20+ (ideally 22)**. Install via `nvm install 22 && nvm use 22` before any scaffold — nothing runs until this is done.
- **Frontmatter round-tripping / formatting drift:** the real files are not uniform — `commands/git.md` starts with `**Purpose**:` and `@include shared/...` lines and has *no* standard `name`/`description` frontmatter, while agents do. `gray-matter`'s `stringify` can reorder keys, change quote style, or normalize whitespace. Test a round-trip on a real file early; if it mangles formatting, fall back to editing only the body and splicing frontmatter surgically, or preserve `raw` and only rewrite when the user actually changes a field.
- **`enabledPlugins` semantics:** the user's `settings.json` currently has **no `enabledPlugins` key**, and plugin enablement is spread across the settings cascade (there's a dedicated `ccplug` CLI managing per-project `enabledPlugins`/`skillOverrides`). Confirm the exact key shape and default behavior (unlisted = enabled?) by inspecting `ccplug` and a project that uses it, or the plugin toggle may write the wrong structure. Writing to the user-level `settings.json` also risks clobbering hand-maintained content — the backup step is mandatory here.
- **Writing to a live, in-use directory:** `~/.claude` is actively read by running Claude Code sessions and background daemons (`daemon.log`, `sessions/`, `history.jsonl` are all live). Restrict writes strictly to `agents/`, `commands/`, `skills/`, and the `enabledPlugins` key — never touch sessions, credentials (`.credentials.json`), or logs. A traversal bug here could corrupt the user's whole Claude setup, so the path guard + backups are load-bearing, not polish.
- **Security exposure:** this app grants HTTP read/write to the home directory; if the dev server binds `0.0.0.0` it's a remote-file-write hole. Bind to `127.0.0.1` and add the local-only middleware (Phase 5) — do not defer it if the port is ever exposed.
- **CodeMirror + SSR:** CodeMirror 6 must be client-only (`<ClientOnly>`) or Nitro SSR will throw on `window`/`document`. Minor but will block Phase 2 if missed.
